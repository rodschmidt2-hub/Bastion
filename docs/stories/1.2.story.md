# Story 1.2 — Gestão de Acesso: Ativar/Desativar Cliente com Histórico

**Epic:** 1 — Cadastro e Ciclo de Vida do Cliente
**Status:** Done
**PRD refs:** FR-03, FR-03b, FR-03c, FR-03d, FR-03e, FR-04

## Descrição
Como gestor, quero ativar e desativar clientes manualmente com motivo obrigatório, e ver o histórico completo de transições de status, para ter controle e rastreabilidade do ciclo de vida de cada cliente.

## Acceptance Criteria

- [ ] Botão "Desativar" no perfil do cliente: abre modal com campo motivo (obrigatório) → muda status para `inativo`
- [ ] Botão "Reativar" no perfil do cliente (visível se status = inativo ou suspenso) → muda status para `ativo`
- [ ] Reativação é sempre manual — quitar fatura não reverte status automaticamente
- [ ] Desativação automática por inadimplência: job `checar-suspensoes` (Story 5.1) muda status para `suspenso`
- [ ] Desativação automática: todos os contratos do cliente marcados como `cancelado` → status muda para `inativo`
- [ ] Toda transição de status registra evento em `eventos_cliente`: status anterior, status novo, motivo, quem fez, quando
- [ ] Seção "Histórico de Status" dentro da tab Dados do cliente exibe timeline de eventos de status em ordem cronológica reversa
- [ ] Listagem de clientes exibe badge de status: verde (ativo), cinza (inativo), vermelho (suspenso)
- [ ] Clientes `suspensos` aparecem em destaque na página `/financeiro` global

## Tarefas Técnicas

- [ ] Server Action: `desativarCliente(id, motivo)` — atualiza status, motivo_inativacao, data_inativacao + registra em eventos_cliente
- [ ] Server Action: `reativarCliente(id)` — atualiza status para ativo + registra em eventos_cliente
- [ ] Server Action: `verificarContratosCliente(id)` — chamada após cancelar contrato: verifica se todos estão cancelados → desativa
- [ ] Componente `desativar-modal.tsx`: campo motivo obrigatório + confirmação
- [ ] Atualizar listagem de clientes: badge de status com cores
- [ ] Componente `historico-status-section.tsx`: timeline de `eventos_cliente` filtrada por tipo de status, exibida na tab Dados
- [ ] Atualizar `/financeiro/page.tsx`: destaque para clientes suspensos

## Notas Técnicas
- Schema sem migration: `clientes` já tem `status`, `motivo_inativacao`, `data_inativacao`, `data_suspensao`
- Status válidos no schema: `ativo` | `inativo` | `suspenso`
- `eventos_cliente` é a tabela de histórico — escrita pela aplicação (não trigger)
- Suspensão automática por inadimplência é responsabilidade do job da Story 5.1 — esta story apenas exibe o resultado
- Desativação por contratos cancelados é trigger de negócio — implementar na Server Action de cancelamento de contrato

## Definição de Pronto
- Gestor consegue desativar e reativar cliente manualmente
- Todo status change registra evento no histórico
- Histórico de status visível na tab Dados do cliente
- Badge de status aparece na listagem
