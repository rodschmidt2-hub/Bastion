# Story 5.1 — Fatura Unificada por Cliente e Inadimplência

**Epic:** 5 — Financeiro e Cobranças
**Status:** Done
**PRD refs:** FR-31, FR-31b, FR-31c, FR-32

## Descrição
Como admin, quero gerar faturas unificadas por cliente/competência com breakdown por contrato, registrar pagamentos vinculados e ter controle de inadimplência com suspensão automática configurável.

## Acceptance Criteria

- [ ] Admin pode gerar fatura manualmente para um cliente: competência (YYYY-MM), data de vencimento, tipo (regular/cortesia)
- [ ] Geração inclui automaticamente todos os `contrato_itens` ativos do cliente na competência — admin não seleciona item a item
- [ ] `saldo_devedor` calculado automaticamente (`valor_total - valor_pago`) — coluna gerada no banco
- [ ] Tab Financeiro do cliente lista faturas: competência, vencimento, valor total, valor pago, saldo, status
- [ ] Status da fatura: pendente → atrasado (vencimento ultrapassado) → pago / parcial
- [ ] Admin registra pagamento vinculado a uma fatura: data, valor, forma de pagamento, comprovante (URL)
- [ ] Pagamento atualiza `valor_pago` da fatura e recalcula `saldo_devedor` automaticamente
- [ ] Fatura com `saldo_devedor = 0` muda status para `pago`
- [ ] Cliente com fatura atrasada por mais de `dias_ate_suspensao` dias → status do cliente muda para `suspenso`
- [ ] Reativação do cliente é **sempre manual** — quitar fatura não reverte status automaticamente
- [ ] Novo contrato sem pró-rata: não gera item na fatura vigente, entra na próxima competência
- [ ] Página `/financeiro` global exibe faturas em aberto por cliente com saldo devedor total

## Tarefas Técnicas

- [ ] Server Action: `gerarFatura(cliente_id, competencia, data_vencimento, tipo)`
- [ ] Server Action: `adicionarItemFatura(fatura_id, contrato_item_id, descricao, valor)`
- [ ] Server Action: `registrarPagamento(fatura_id, data, valor, forma_pagamento, comprovante_url)`
- [ ] Server Action: `marcarFaturasAtrasadas(agencia_id)` — muda status `pendente → atrasado` quando `data_vencimento < hoje` (chamada por job diário)
- [ ] Server Action: `verificarSuspensao(agencia_id)` — atualiza status de clientes com fatura atrasada > `dias_ate_suspensao` (chamada por job)
- [ ] Componente `financeiro-tab.tsx`: lista de faturas com status badge + botão registrar pagamento
- [ ] Componente `fatura-drawer.tsx`: detalhe da fatura com itens + pagamentos registrados
- [ ] Componente `pagamento-modal.tsx`: form de registro de pagamento vinculado à fatura
- [ ] Atualizar `/financeiro/page.tsx`: painel global de faturas em aberto com filtro por status

## Notas Técnicas
- Schema completo já existe: `faturas`, `fatura_itens`, `pagamentos` — sem migration necessária
- `faturas.status` válidos: `pendente`, `parcial`, `pago`, `atrasado`, `suspenso`, `cancelado`, `em_acordo`, `liquidado_por_acordo`, `cancelado_por_distrato`
- `clientes.status` válidos: `ativo`, `inativo`, `suspenso`
- `dias_ate_suspensao` é por cliente (default 30) — configurável via campo no cadastro do cliente
- `inadimplencia_snapshot` é materialized view já existente — usar para página global `/financeiro`
- Job `checar-suspensoes` já previsto em `jobs_log` — `verificarSuspensao` será chamada por Vercel Cron

## Definição de Pronto
- Admin gera fatura manual com itens e registra pagamento
- `saldo_devedor` atualizado automaticamente após pagamento
- Cliente com fatura atrasada > threshold entra em `suspenso`
- Página `/financeiro` exibe visão consolidada de inadimplência
